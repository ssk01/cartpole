# CartPole 游戏实现解析

---

## 这个游戏是什么

一根杆子立在一辆小车上，你用左右方向键推小车，目标是不让杆倒、不让车跑出平台。就这么简单。

打开 `cartpole_game.html` 就能玩。

---

## 核心问题：4 个数字 + 1 个动作 → 下一个状态

整个游戏的物理引擎就是一个函数：**给定当前 4 个状态值和你的操作，算出下一瞬间的 4 个状态值。**

### 4 个状态（state）

| 变量 | 含义 | 你能感知到的 |
|------|------|------------|
| `x` | 小车在画面上的像素位置 | 车在左边还是右边 |
| `xDot` | 小车速度 | 车在往哪边跑、跑多快 |
| `theta` (θ) | 杆的角度（弧度，顺时针为正） | 杆往哪边歪 |
| `thetaDot` (ω) | 杆的角速度 | 杆在往哪边倒、倒多快 |

### 1 个动作（action）

| 值 | 含义 |
|----|------|
| -1 | 向左推（按左方向键） |
| 0 | 不推（松开所有键） |
| +1 | 向右推（按右方向键） |

注意这个游戏允许"不推"，而 Gymnasium 的 CartPole 只有推左/推右两个选项——这是游戏版比训练版更好玩的原因之一。

### 物理计算：`doStep()` 函数

这是整个游戏的核心，20 行代码决定一切：

```javascript
function doStep() {
    var cos = Math.cos(theta);
    var sin = Math.sin(theta);
    var F = throttle * strength;     // 你的推力：±15 或 0
    
    // 联立方程求解小车加速度和杆角加速度
    var det = (l * cos**2 - l * (1 + Mm));
    var xDotDot = (-l*F + l*l*sin*(thetaDot**2) - g*l*sin*cos) / det;
    var thetaDotDot = (-cos*F + l*cos*sin*(thetaDot**2) - g*sin*(1+Mm)) / det;
    
    // 摩擦力（让车不会永远滑下去）
    xDotDot -= friction * xDot;

    // 欧拉积分：加速度 → 速度 → 位置
    x += xDot * dt;            // 位置 = 旧位置 + 速度 × 时间步
    theta += thetaDot * dt;    // 角度 = 旧角度 + 角速度 × 时间步
    xDot += xDotDot * dt;      // 速度 = 旧速度 + 加速度 × 时间步
    thetaDot += thetaDotDot * dt; // 角速度 = 旧角速度 + 角加速度 × 时间步
    t += 1;
}
```

**用人话说这段代码做了什么：**

1. **算推力**：你按了什么键 → 力 F = ±15 或 0
2. **算加速度**：根据牛顿力学（拉格朗日方程），从当前角度、角速度、推力，算出小车加速度 `xDotDot` 和杆角加速度 `thetaDotDot`
3. **积分更新**：加速度 → 更新速度 → 更新位置/角度

这就是物理学里的因果链：**力 → 加速度 → 速度 → 位置**。

### 为什么杆会倒？

看角加速度的公式：

```
thetaDotDot = (-cos*F + l*cos*sin*(thetaDot²) - g*sin*(1+Mm)) / det
```

关键项是 `-g*sin*(1+Mm)`：重力拉着杆往它倾斜的方向**加速倒下**。θ 越大，sin(θ) 越大，倒得越快。这就是为什么一旦杆开始偏，不纠正就会越来越快地倒下——**正反馈**。

### 为什么推车能救杆？

另一个关键项是 `-cos*F`：你推车产生的力会让杆往**反方向**加速。杆往右倒 → 你往右推车 → 车突然加速往右 → 杆因为惯性相对车身往左"回正"。

**这就是控制的本质：用一个你能控制的量（推力），去对抗一个你不能直接控制的量（重力引起的倒塌）。**

## 游戏循环怎么跑的

```
每秒 100 帧（framesPerSec = 100）
  └── 每帧跑 50 个物理步（stepsPerFrame = 50）
        └── 每步 dt = 0.001 秒

所以：1 帧 = 50 × 0.001 = 0.05 秒游戏时间
      1 秒真实时间 = 100 帧 = 5 秒游戏时间
```

每帧做的事：
1. 跑 50 步物理（调用 50 次 `doStep()`）
2. 如果是自动模式，调用 `pickAction()` 决定下一帧的动作
3. 检查是否出界（`cartPoleInBounds()`）
4. 画画面（`paintCanvas()`）
5. 更新状态面板（`updateStatePanel()`）

### 为什么不是每步都画？

因为 `dt = 0.001` 太小了——如果每 0.001 秒画一帧，需要 1000fps，屏幕根本刷不过来。所以攒 50 步画一帧，物理精度高但渲染负担低。

## 死亡条件

```javascript
function cartPoleInBounds() {
    var poleTipX = x - poleLength * Math.sin(theta);
    var poleTipY = platformY - cartHeight - poleLength * Math.cos(theta);
    return (
        x > platformLeft - cartWidth / 2       // 车没跑出左边
        && x < platformRight + cartWidth / 2    // 车没跑出右边
        && poleTipY < platformY                 // 杆顶没碰到地面
        && poleTipX > 0 && poleTipX < canvas.width  // 杆顶没超出画面
    );
}
```

注意：这个游戏**没有角度限制**——杆可以转过水平线甚至转一整圈（如果你勾选 Immortal 模式可以观察到）。Gymnasium 版的限制是 ±12°，严格得多。

## 理论上能一直玩下去吗？

**能。** 这就是为什么它是经典问题。

倒立摆在控制论里有一个已知的**解析解**——PID 控制器可以无限期稳定杆。实际上你在自动模式看到的那个简单策略：

```javascript
if (theta > Math.PI / 50) return LEFT_PUSH;
else if (theta < -Math.PI / 50) return RIGHT_PUSH;
else return NO_PUSH;
```

这个"杆往哪边倒就往哪边推"的策略，在 Immortal 模式下可以跑很久（勾选 Immortal 试试）。但它不是最优的——因为它不看 `x`（车的位置），所以车会慢慢飘到边缘最终掉出去。

一个完美的策略需要同时考虑**所有 4 个状态变量**：不仅让杆不倒，还要让车保持在中间。这就是为什么 4 个状态变量缺一不可——少了任何一个信息，你都无法做出最优决策。

## 为什么它是强化学习/控制论的经典问题

### 1. 简单但不平凡

状态只有 4 个数字，动作只有 2-3 个选择。但要解好并不容易——需要平衡短期（不让杆倒）和长期（不让车飘走）两个目标。

### 2. 不稳定平衡

杆直立是一个**不稳定平衡点**——任何微小扰动都会让杆倒下。这和稳定平衡（如球在碗底）本质不同。不稳定系统需要**持续的、精确的控制**才能维持，这正是控制论要解决的核心问题。

### 3. 延迟后果

你现在推一下车，杆不会立刻倒或立刻稳——效果要经过几步物理演化才显现。这种**延迟后果**正是强化学习要处理的难题（回忆 DQN 里 Q 值需要从 reward 逐步回传）。

### 4. 连续 vs 离散的桥梁

状态是连续的（4 个实数），动作可以是离散的（左/右）——这让它同时适合测试连续控制和离散决策算法。

### 5. 有解析解可对照

控制论已经有了最优解（LQR 控制器），所以可以用来**验证**强化学习算法是否学到了接近最优的策略。如果 DQN 在这个问题上都学不好，那在更难的问题上就更没戏了。

### 6. 10 秒就能跑完一局

训练快——不像 Atari 要跑几个小时。适合快速迭代、调试算法。

## 自定义代码功能

游戏保留了原版的一个杀手功能：你可以在页面上**直接写 JavaScript 代码**来控制小车。

切换到"自动"模式后，会出现一个代码编辑框。默认的控制逻辑是：

```javascript
if (theta < -Math.PI / 50) { return RIGHT_PUSH; }
else if (theta > Math.PI / 50) { return LEFT_PUSH; }
else { return NO_PUSH; }
```

你可以改成任何逻辑。比如一个更好的策略（同时考虑角度和位置）：

```javascript
// 加权组合：角度权重大（紧急），位置权重小（长期）
var signal = theta * 50 + thetaDot * 5 + (x - 200) * 0.1 + xDot * 0.5;
if (signal > 0) return LEFT_PUSH;
else if (signal < 0) return RIGHT_PUSH;
else return NO_PUSH;
```

点击"运行代码"就能看到效果。这就是**在浏览器里手动实现一个控制策略**——DQN 做的事情和你做的一样，只不过它用神经网络自动找到 `signal` 的权重，而不是你手动调。

## 游戏版 vs Gymnasium 训练版的对比

| | 游戏版（本文件） | Gymnasium CartPole-v1 |
|--|---------------|---------------------|
| 物理常数 | g=10, l=200px, strength=15 | g=9.8, l=0.5m, force=10N |
| 时间步 | dt=0.001, 每帧50步 | dt=0.02, 每步就是一帧 |
| 动作 | 左/右/不推（3个） | 左/右（2个） |
| 死亡条件 | 杆碰地面或车出平台 | \|θ\|>12° 或 \|x\|>2.4m |
| 摩擦力 | 有（0.15阻尼） | 无 |
| 用途 | 人玩、理解物理 | RL 算法训练 |

游戏版更"好玩"是因为：有摩擦（车会自然减速）、有"不推"选项、死亡条件更宽松、物理步更细（手感更平滑）。
